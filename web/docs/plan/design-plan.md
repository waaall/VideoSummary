# 历史任务体验设计方案

## 命名与定位
- **执行监控 → 历史任务**（首选名称）
- 定位原则：
  - **快速开始页负责“创建任务 + 运行态监控”**
  - **历史任务页负责“查看完成结果 + 历史检索”**
  - 页面职责清晰，避免“既要找任务又要监控”的混合体验

## 核心设计原则
- **让用户不必记住 Job ID**：Job ID 作为技术标识，不作为主要检索入口。
- **流程闭环**：创建任务后能一键进入历史任务并定位到该任务。
- **状态清晰**：运行中、完成、失败等状态有明确的视觉差异与行动入口。
- **可找回**：历史结果列表 + 本地持久化，减少“任务丢失”的焦虑。

## 关键用户流程
1) 快速开始提交任务 → 进入运行态监控（同页）
2) 任务完成 → 提示“已完成”，并提供 **查看结果** CTA
3) 点击 CTA → 进入历史任务，自动定位到该任务
4) 用户直接进入历史任务 → 默认展示最近任务列表，可搜索/筛选

## 快速开始页：运行态监控（仅运行时）
**目标：减少跳转，让用户在提交页即可看到进度**
- 运行中显示：状态、进度、耗时、简短日志/阶段提示
- 完成后显示：
  - 任务状态“已完成”
  - **查看结果**按钮（主 CTA）
  - 轻量摘要预览（可选）
- 失败时显示：错误信息 + 重试/重新提交入口
- 可见的 Job ID + 复制按钮（辅助功能）

## 历史任务页：信息结构
**页面布局：左侧历史列表 + 右侧结果显示（Markdown）**

### 左侧：历史结果列表
- 默认展示最近 5–10 条任务（按时间倒序）
- 列表条目内容建议：标题/来源 URL、时间、状态标签、时长
- 搜索入口：
  - 主要：URL / 标题 / 时间
  - 辅助：Job ID（作为高级入口）
- 筛选入口（可选）：状态（完成/失败/运行中）

### 右侧：结果显示区
- 顶部：状态标签、任务信息（时间、耗时、来源）
- 主体：Markdown 结果展示
- 操作按钮：复制/导出/重新生成（视功能而定）

### 非完成状态的处理
- 若选中的任务 **尚未完成**：
  - 显示当前状态与提示
  - 提供 **继续监控** 或 **回到快速开始页** 的入口
  - 避免“只有提示无动作”的空转体验

## 历史结果的存储策略
- **本地持久化**（localStorage 或 indexedDB）
  - 创建任务时记录（job_id、cache_key、来源 URL、时间）
  - **job_id 是一次执行尝试**，用于轮询与查看详情
  - **cache_key 是缓存身份**，同一输入会复用同一个 cache_key
  - 若同 cache_key 出现新 job_id，仅保留最新一次尝试记录
  - 任务完成时更新状态与摘要
- 若未来有账号系统，可扩展为服务端历史记录
- 历史任务优先展示本地历史，保证跨页面可找回

## 空状态与异常状态
- 空状态（无历史记录）：提示用户去快速开始创建任务
- 失败状态：错误信息 + 重试入口
- 过期状态（若有）：提示结果过期与重新生成选项

## 体验要点总结
- 快速开始页 = **运行态监控 + 任务创建**
- 历史任务 = **完成结果查看 + 历史检索**
- Job ID **存在但不强迫记忆**
- 主流程始终有明确 CTA，避免“提示无动作”的断点体验
